1. Rewrite `apps/backend/src/modules/dashboard/dashboard.service.ts`:
   - Make `getKpis()` async: `async getKpis(): Promise<DashboardResponse>`
   - Await all service calls. Use `Promise.all()` for parallel execution since the 5 getStats calls and findAll are independent:
     ```typescript
     const [salesStats, orderStats, inventoryStats, productStats, purchaseStats, allOrders] =
       await Promise.all([
         this.salesService.getStats(),
         this.ordersService.getStats(),
         this.inventoryService.getStats(),
         this.productsService.getStats(),
         this.purchasesService.getStats(),
         this.ordersService.findAll({ limit: 5, sort: '-createdAt' }),
       ])
     ```
   - Rest of the response assembly logic stays the same — map recentOrders from allOrders.data.slice(0, 5), map lowStockItems from inventoryStats.lowStockItems.slice(0, 5)
   - Keep the DashboardResponse, DashboardStats, LowStockItem, RecentOrder interfaces at the top
   - The response shape MUST be identical to the current implementation — this is what both web and mobile frontends consume

CRITICAL (RESEARCH.md Pitfall 2): Since all service methods are now async, forgetting `await` will return Promise objects instead of data. The `Promise.all()` pattern ensures all are awaited correctly.

2. Update `apps/backend/src/modules/dashboard/dashboard.controller.ts`:
   - Make `getKpis()` async: `async getKpis()` and `return await this.dashboardService.getKpis()`
   - NestJS handles Promise returns transparently, but making the controller explicitly async is cleaner

NOTE: The `allOrders.findAll({ limit: 5, sort: '-createdAt' })` now queries the DB. Passing `limit: 5` directly means the DB only returns 5 rows (efficient). The `sort: '-createdAt'` must be parsed by the orders service sort logic.
